1. Introduction

1.1 Purpose

The aim of this project is to digitize existing documents/files via scanning. The converted digital format will be stored into storage using a simple document system capable search and retrieval for attaching the same.

1.2 Document Conventions

The project is a government-based initiative. The project has a user management system while controlling their own and public documents in a digitalized form. Users have different ranking and control over the documents accordingly.

1.3 Intended Audience and Reading Suggestions

This software is intended for:

  • developers who can modify the project and can add more features.
  • users such as employees, manager can read the document and can take part in editing the exiting document.
  • administrator can overview the documents as well as can change as needed.
  • to read the document, users have to log in. Some users can edit the document, if the user has specific role.

1.4 Product Scope

This document digitization service will help employees and users to get information in a faster way. Being user friendly and efficient, this application will benefit the organization. this documentation system will provide a substantial service in further business strategies

1.5 References

Ref 1: Scrimba.com

2. Overall Perspective

2.1 Product Perspective

This program has many features which will help people to store their documents later they can restore their document as per needed. This will contain and store all the previous documents which was present in the file. People who have account in this can access the document and they can add, remove and edit information from this. User can create an account and from that they can access the document. Also, there will be multiple type of users as example: public and private.

2.2 Product Functions

The major functionalities of the system are described below:

  • Two types of Users Information: Official and Non-Official.
  • All users will have unique ID and password for logging on to the account's information.
  • Store Data and Access Documents: User can store Documents and if needed they can access the documents.
  • Search, Edit, Remove and Archive: User can search any specific documents by date or name. Document can be edited and if you want you can remove your Documents. After a specific period of time your document you be archived.
  • Hardware Accessibility: User can access their data from different platform like PCs, Laptops or even Smartphones.
  • Optimized Versions: Available in android version which user can use to access their files.
  • Scan and Read: This feature will help the user to scan and read the Document from it.
  • System Management: User will get a notification in the mail after Login. Accounts with 3 failed login attempts should block and User has to send a reset request to access the account.
  • The solution provides very simple and modified features which are very easy to view and operate various features

2.3 User Classes and Characteristics

We can find various users for this system. People with different capabilities and authorizations can have access to this system Public/non-official user: Normal people can have an account and will have access of basic features of the system. Official / corporate user: Employees and managers are known as official user. They may have access to edit the documents. Admin user: People with highest authority belong to this sort of user. They have the power to make necessary changes. Software developers: Developers are capable of to make changes as per request of the authority. They can add features and extend projects source code

2.4 Operating Environment

This system should be able to run on Windows and Android.

2.5 Design and Implementation Constraints

Design and Implementation Constraints This system is designed in Django framework with firebase database. The language constraint of this system is only English. The system is limited to HTTP/HTTPS protocols.

2.6 User Documentation

The system is designed to be intuitive and user friendly. But if there are further queries, the user can go through the user manual and online video tutorials which will be provided later.

2.7 Assumptions and Dependencies

The user must have proper knowledge about how to use the computer or mobile phone as well as to properly scan the documents for upload. The program requires to be working via online for OCR and uploading features. Document sharing functionalities also require internet connection.

3. External Interface Requirements

3.1 User Interfaces

In this user interface, user will have a profile where he can have the access of the documents where he can view, edit and delete his documents. He will also have a notification bar where he can see the documents people are uploading throughout the time. He will also have the access to only view the document of the public folders documents. Lastly, he can also able to upload his documents.

3.2 Hardware Interfaces

For the hardware interface we will need to use the Android app. For Laptop or Computer, the version (works with a browser) will be [HTTP/HTTPS]

3.3 Software Interfaces

For the webpage version of the product, we are using HTML, CSS, JavaScript and Bootstrap. Our database is in Firebase. Python and Django are used for the backend. For making android version of this product, we will use Kotlin.

3.4 Communications Interfaces

We will use Email and Google login for user authentication. For communication protocols, HTTP or HTTPS will be used and JSON will bs used as Rest API. Since we are using Firebase for the database, passwords will be encrypted in Firebase.

5. Other Non-Functional Requirements

5.1 Perfomance Requirement

This will help to understand the usability of the website and know about how it works. After signing in to the account the user has time to do the work he wants to do that. It will take time to upload the document.

Response Time: For systems that have to support significant numbers of users the cost of response times delays can actually be measured in monetary terms and therefore can form part of trade-off studies between different architectures providing different levels of performance.

5.2 Safety Requirements

System use shall not cause any harm to users. It will not break any political and government rule. We respect our user’s privacy and are committed to protecting their personal documents. Backup, recovery and business continuity should ensure adequate backup of data as may be required by their operations. Document should be backed up periodically. An offsite backup is necessary for recovery from major failures / disasters to ensure business continuity.

5.3 Security Requirement

Unauthorized access to the system and its data is not allowed. System use secured database. Account ID and password (PIN) Requirements Sign-off Button and Failed Login Attempts.

5.4 Software Quality Attributes

  • Adaptability: To adapt the way of the this works on different platforms.
  • Correctness: The application should be correct in terms of its functionality, calculations used internally and the navigation should be correct.
  • Flexibility: Should be flexible enough to modify. Adaptable to other products with which it needs interaction. Should be easy to interface with other standard 3rd party components.
  • Interoperability: Interoperability of one system to another should be easy for the product to exchange data or services with other systems. Different system modules should work on different operating system platforms, different databases, and protocol conditions.
  • Maintainability: The maintainability of a software system depends on its: Readability, Extensibility and Testability.
  • Portability: This can be measured in terms of Costing issues related to porting, Technical issues related to porting, Behavioral issues related to porting.
  • Reliability: The behavior over time for the fulfillment of a given specification depends on the reliability of the software system.
  • Reusability: Software reuse is a good cost-efficient and time-saving development way. Different code library classes should be generic enough to use easily in different application modules. Dividing the application into different modules so that modules can be reused across the application.
  • Testability: The system should be easy to test and find defects. If required should be easy to divide into different modules for testing.
  • Usability: This can be measured in terms of ease of use. The application should be user-friendly. Should be easy to learn. Navigation should be simple. The system must be:
    • Easy to use for input preparation, operation, and interpretation of the output.
    • Provide consistent user interface standards or conventions with our other frequently used systems.
    • Easy for new or infrequent users to learn to use the system.

5.5 Business Rules

One user can have only one account. Users are not allowed to see other account's information unless that information is uploaded publicly. All search history will be saved and admin can see those.
